# Demo 笔记：多 Agent 项目管理面板

## 取舍

### 选择静态 Demo 的原因

- **部署简单** — 不需要后端，GitHub Pages 直接 serve
- **演示友好** — 任何浏览器打开 index.html 即可
- **可分享** — 一个 URL 即可分享给任何人
- **维护成本低** — 无服务器、无数据库、无 API 调用

### 选择模拟数据的考量

- **避免泄漏** — 不展示真实项目状态（含内部信息）
- **避免依赖** — 不依赖真实 API 权限、机器状态
- **保持一致** — 演示效果稳定，不因真实数据变化而变化
- **便于扩展** — 数据结构清晰，未来替换为真实数据简单

## 设计决策

### 卡片字段选择

每个项目卡片显示 10 个字段：name / agent / machine / phase / status / risk / latest_commit / latest_report / blocker / next_action。

**为什么是这 10 个：**
- 5 个核心字段（name/agent/machine/phase/status）—— 识别项目
- 2 个状态字段（risk/blocker）—— 识别问题
- 2 个进度字段（latest_commit/latest_report）—— 跟踪历史
- 1 个行动字段（next_action）—— 推动下一步

### 筛选按钮设计

6 个筛选按钮：All / Active / Blocked / Done / Needs Review / High Risk。

**为什么是这 6 个：**
- 覆盖所有 status 枚举
- 加上 High Risk 用于风险维度的快速筛选
- 符合"按状态和风险筛选"的核心需求

### 视觉风格

采用 Notion 风格：白色背景、圆角卡片、低饱和度配色、清晰的层级结构。

**为什么选择 Notion 风格：**
- 项目负责人熟悉的视觉语言
- 信息密度合适，不密集也不稀疏
- 移动端可读性好

## 限制

### 本 Demo 的限制

1. **静态数据** — 不会自动更新，需要手动替换 data.json
2. **无 API 集成** — 不会连接 GitHub API 拉取真实 commit
3. **无历史快照** — 不保存历史状态，无法看趋势
4. **无通知机制** — blocked 不会主动提醒
5. **无调度建议** — 不会主动建议下一步该做什么

### 真实系统会遇到的问题

1. **报告格式不统一** — 不同 agent 报告格式不同，标准化复杂
2. **commit 解析不稳定** — commit message 格式多样，解析需要容错
3. **agent 状态不一致** — 同一项目可能多个 agent 报告不同状态
4. **数据延迟** — 状态更新有延迟，可能错过关键变化
5. **权限问题** — 读取 GitHub、报告目录可能需要权限

## 演进路径

从静态 Demo 到真实系统的路径：

1. **第一阶段（静态 Demo）：** 当前状态
2. **第二阶段（半自动）：** 数据采集脚本 + 手动触发更新
3. **第三阶段（自动）：** GitHub API + 报告解析 + 定时刷新
4. **第四阶段（智能）：** 历史快照 + 趋势分析 + 调度建议
5. **第五阶段（生态）：** Telegram bot + 自动派单 + 跨平台集成

## 经验总结

- 静态 Demo 是验证想法的最快方式
- 数据结构设计比界面设计更重要
- 字段命名要"自我解释"
- 标注限制比假装完美更专业
