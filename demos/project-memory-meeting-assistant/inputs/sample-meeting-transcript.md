# 会议转写：TeamFlow Dashboard 项目周会（2026-06-12）

## 会议内容

### 议题 1：MVP 进度回顾

**李明：** 我们先看一下上周的进度。王芳，前端看板做得怎么样了？

**王芳：** 基础看板 UI 已经出来了，可以拖拽任务，但还不能保存状态。预计本周能完成数据对接。

**赵强：** 后端 API 已经完成 70%，任务 CRUD 接口都通了。但权限接口还没做，我们在等今天的决策。

### 议题 2：客户反馈与风险提醒

**李明：** 我上周跟两个潜在客户聊了一下，他们都有一个共同需求：希望看到风险状态。比如哪个任务延期了、哪个需求有变更风险。他们觉得现在的看板只有任务状态，看不到风险。

**张伟：** 这个需求有价值，但风险追踪需要设计风险登记和评估机制。如果放进 MVP，会增加复杂度。

**王芳：** 风险功能我可以先做最简单的：每个任务加一个"风险等级"标签（低/中/高），用颜色区分。不需要复杂的风险登记流程。

**李明：** 这个方案可行。但我们需要确认：风险等级谁来标注？是任务创建者自己标，还是项目经理统一标？

**张伟：** 我的建议是：创建者自己标，但项目经理可以修改。这样不会增加创建者负担，也能保证准确性。

### 议题 3：权限系统复杂度

**赵强：** 关于权限系统，我想提一下。按照之前的设计，权限系统包含角色管理、资源权限、操作权限、数据权限，这个复杂度大概需要 2-3 周。如果放进 MVP，我们的发布时间要推迟。

**张伟：** 我同意赵强的评估。我们上周讨论过简化权限，但一直没有明确结论。今天需要做个决策。

**李明：** 我的建议是：MVP 只做两个角色——管理员（能看所有内容）和成员（能看自己参与的任务）。不做复杂的 RBAC。用户管理用邀请制，不需要注册流程。

**王芳：** 这个简化我同意。前端工作量可以减少 60%。

**赵强：** 后端也可以减少很多。我可以把权限逻辑写在业务层，不需要独立的权限服务。

**张伟：** 好，那我们今天明确决策：MVP 权限系统只做管理员/成员两个角色，用邀请制。复杂的 RBAC 放到 Phase 2。

### 议题 4：Roadmap 更新

**李明：** 基于今天的讨论，我提议更新 Roadmap：

- 风险提醒（简化版）加入 MVP
- 权限系统（复杂版）延后到 Phase 2
- 增加一个目标：6 月底前给 2-3 个潜在客户试用只读看板

**张伟：** 只读看板是什么意思？

**李明：** 就是我们先把看板做出来，不给客户编辑权限，让他们只看、评论。这样可以收集反馈，同时不需要做复杂权限。

**陈静：** 这个方案好，我可以快速做一版只读看板的 UI。

### 议题 5：待确认问题

**李明：** 有几个问题需要确认：

1. 风险标签的触发规则是什么？是手动标还是自动标（比如任务延期自动标高风险）？
2. 只读看板给哪些客户试用？我们需要筛选标准。
3. 客户试用期间的反馈收集方式：用问卷还是直接访谈？

**张伟：** 我建议：第一个版本手动标风险。自动规则需要数据分析能力，放在 Phase 2。试用客户筛选标准：团队规模 10-30 人、已经在用类似工具但不满意、愿意给反馈。

**李明：** 好，这些待确认问题我会整理后发到群里，大家各自领域的问题负责确认。

## 决策记录

3. **Roadmap 更新** — 增加目标：6 月底前给 2-3 个潜在客户试用只读看板
