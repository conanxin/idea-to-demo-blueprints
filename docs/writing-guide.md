# 写作指南：如何把一个想法整理成可落地的 Demo 方案

> 本指南帮助你将一个模糊的想法转化为其他人可以照着执行的 Blueprint。

---

## 核心原则

### 1. 保留原始想法

不要删除原始想法。即使它听起来很口语化、很随意，也要保留。原始想法是方案的灵魂，它记录了当时的灵感、场景和情绪。后续的方案是对这个灵感的结构化展开，不是替代。

### 2. 从问题出发，不是从解决方案出发

先描述痛点：
- 现在是怎么做的？
- 哪里不舒服？
- 浪费了多少时间？
- 重复了多少次？

然后才是：这个想法如何解决它。

### 3. 最小可行版本（MVP）优先

不要写"理想状态"的方案。写"花 1-2 天就能验证"的方案。理想状态的方案永远不会被执行，因为资源不够。MVP 方案可以被验证，验证后的改进才有方向。

### 4. 假设读者什么都不知道

方案的读者可能是：
- 另一个团队的人
- 3 个月后的你自己
- 一个刚入职的工程师
- 一个非技术背景的产品经理

不要假设读者知道你的项目上下文。把上下文写清楚。

### 5. 提供可直接复制粘贴的内容

如果方案涉及：
- 代码 → 提供完整的代码块
- Prompt → 提供可以直接复制的文本
- 配置 → 提供完整的配置示例
- 命令 → 提供可以在终端直接执行的命令

读者不应该需要自己去"补充"什么。

## 写作流程

### 步骤 1：记录原始想法（5 分钟）

1. 打开一个空白文档
2. 写下这个想法最初是怎么出现的
3. 保留原始语境：谁、在哪里、发生了什么、为什么这个想法冒出来
4. 不要改写，不要润色，直接复制粘贴聊天记录或语音转写

### 步骤 2：定义问题和场景（15 分钟）

1. 描述当前的工作流程（现状）
2. 列出具体的痛点（用数字或时间量化更好）
3. 说明为什么这个问题值得解决（影响范围、频率、代价）

### 步骤 3：方案概述（10 分钟）

用 2-3 段话描述核心方案。不要展开技术细节，先回答：
- 这个方案是什么？
- 它怎么做？
- 为什么它比现有方案好？

### 步骤 4：设计 MVP（30 分钟）

1. 定义输入：需要什么？数据、工具、权限、环境
2. 定义输出：验证成功时应该看到什么？
3. 定义验证标准：怎么判断这个 Demo 做成功了？
4. 列出具体步骤：每一步做什么，用什么工具，预计多久

### 步骤 5：技术架构（20 分钟）

1. 列出最小技术栈（只选必需的，不要堆砌）
2. 描述组件和它们的关系（用文字或简单 ASCII 图）
3. 说明数据/信息如何流动

### 步骤 6：Prompt 模板（10 分钟）

如果方案涉及 AI Agent，提供一个可以直接复制给 Agent 的 Prompt 模板。这个模板应该：
- 包含角色定义
- 包含输入格式
- 包含输出格式
- 包含约束条件
- 包含质量要求

### 步骤 7：验收标准（10 分钟）

- 功能：核心功能是否工作？
- 质量：输出是否达到可接受标准？
- 性能：是否在合理时间内完成？
- 可用性：是否可以直接使用，不需要额外配置？

## 质量标准检查清单

在提交方案前，检查以下项目：

- [ ] 原始想法被完整保留，没有被改写或删除
- [ ] 问题描述具体，包含场景和量化数据
- [ ] MVP 可以在 1-2 天内验证
- [ ] 所有步骤具体到可以执行（不需要读者"补充"）
- [ ] 提供了可直接复制粘贴的代码/Prompt/配置
- [ ] 技术架构图用文字描述清楚，没有模糊地带
- [ ] 验收标准可验证，不是"感觉好"
- [ ] 没有遗漏关键步骤（权限、配置、环境准备）
- [ ] 语言简洁，没有废话
- [ ] 适合技术背景和非技术背景的读者阅读

## 常见错误

| 错误 | 表现 | 修正 |
|------|------|------|
| 缺少原始想法 | 方案看起来是"设计出来的"，不是"从真实需求出发的" | 补充原始想法段落 |
| 问题描述模糊 | "效率不高"、"体验不好" | 改为"每次手动操作需要 15 分钟，每天重复 5 次" |
| MVP 过大 | 需要 2 周以上才能验证 | 缩减到核心假设，1-2 天验证 |
| 步骤不具体 | "配置好环境" | 改为"运行 `pip install -r requirements.txt`" |
| 缺少 Prompt | 方案提到 AI 但没有给 Prompt 模板 | 补充可复制的 Prompt |
| 没有验收标准 | 不知道"做完了"是什么意思 | 定义明确的标准，如"输出包含 10 个字段" |
| 过度技术化 | 堆砌技术术语，没有解释 | 假设读者不懂，解释每个技术选择的原因 |

## 格式规范

### 文件命名

- Markdown 版：`docs/{slug}.md`
- HTML 版：`public/blueprints/{slug}.html`
- 数据条目：`data/blueprints.json`

### slug 规则

- 全部小写
- 单词用连字符分隔
- 不要包含数字序号（用 `project-memory-meeting-assistant` 而不是 `001-project-memory`）
- 保持简洁，不超过 5 个单词

### 图片/图表

- 优先使用文字描述和 ASCII 图
- 如需要图片，放在 `public/assets/` 目录下
- 使用相对路径引用
