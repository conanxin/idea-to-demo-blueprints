# 写作指南：如何把一个想法整理成可落地的 Demo 方案

> 本指南帮助你将一个模糊的想法转化为其他人可以照着执行的 Blueprint。

---

## 核心原则

### 1. 保留原始想法

不要删除原始想法。即使它听起来很口语化、很随意，也要保留。原始想法是方案的灵魂，它记录了当时的灵感、场景和情绪。后续的方案是对这个灵感的结构化展开，不是替代。

### 2. 从问题出发，不是从解决方案出发

先描述痛点：
- 现在是怎么做的？
- 哪里不舒服？
- 浪费了多少时间？
- 重复了多少次？

然后才是：这个想法如何解决它。

### 3. 最小可行版本（MVP）优先

不要写"理想状态"的方案。写"花 1-2 天就能验证"的方案。理想状态的方案永远不会被执行，因为资源不够。MVP 方案可以被验证，验证后的改进才有方向。

### 4. 假设读者什么都不知道

方案的读者可能是：
- 另一个团队的人
- 3 个月后的你自己
- 一个刚入职的工程师
- 一个非技术背景的产品经理

不要假设读者知道你的项目上下文。把上下文写清楚。

### 5. 提供可直接复制粘贴的内容

如果方案涉及：
- 代码 → 提供完整的代码块
- Prompt → 提供可以直接复制的文本
- 配置 → 提供完整的配置示例
- 命令 → 提供可以在终端直接执行的命令

读者不应该需要自己去"补充"什么。

### 6. 必须构建真实 Demo

**这是本方案库的核心要求。** 每个 Blueprint 不能只写方案，必须：
- 构建一个真实可运行的最小 Demo
- 记录完整的构建过程
- 提供关键 Prompt
- 提供复现步骤
- 提供验收标准

## 从想法到 Demo 的四步法

```
捕捉想法 → 设计方案 → 构建最小 Demo → 记录复现过程
```

### 步骤 1：捕捉想法（5 分钟）

1. 打开一个空白文档
2. 写下这个想法最初是怎么出现的
3. 保留原始语境：谁、在哪里、发生了什么、为什么这个想法冒出来
4. 不要改写，不要润色，直接复制粘贴聊天记录或语音转写

### 步骤 2：设计方案（30 分钟）

1. **定义问题和场景**（15 分钟）
   - 描述当前的工作流程（现状）
   - 列出具体的痛点（用数字或时间量化更好）
   - 说明为什么这个问题值得解决（影响范围、频率、代价）

2. **方案概述**（10 分钟）
   - 用 2-3 段话描述核心方案
   - 回答：这个方案是什么？它怎么做？为什么它比现有方案好？

3. **设计 MVP**（5 分钟）
   - 定义输入：需要什么？数据、工具、权限、环境
   - 定义输出：验证成功时应该看到什么？
   - 定义验证标准：怎么判断这个 Demo 做成功了？

### 步骤 3：构建最小 Demo（2-4 小时）

1. 准备输入材料（原始想法、背景文档、会议转写等）
2. 编写关键 Prompt（驱动 Agent 生成 Demo Brief）
3. 让 Agent 基于 Brief 生成任务计划
4. 让 Agent 构建最小原型（HTML/CSS/JS 或其他）
5. 人工检查原型是否符合预期
6. 记录构建过程（遇到的问题、解决方案）

### 步骤 4：记录复现过程（30 分钟）

1. 创建 Demo Pack 目录结构
2. 保存输入材料到 `inputs/`
3. 保存关键 Prompt 到 `prompts/`
4. 保存生成结果到 `outputs/`
5. 保存可运行 Demo 到 `app/`
6. 编写验收清单到 `validation/`
7. 编写复现说明到 `README.md`
8. 更新 `data/blueprints.json` 元数据（含 `demo_url`）

## 如何判断一个想法是否适合进入方案库

一个想法要进入方案库，必须满足以下条件：

| 判断标准 | 说明 | 必须满足？ |
|---------|------|----------|
| **有真实问题** | 这个想法要解决一个具体的、可描述的问题 | ✅ 必须 |
| **有明确使用场景** | 有人在真实工作中会遇到这个场景 | ✅ 必须 |
| **能做 Demo** | 可以在 1-3 天内构建一个可验证的原型 | ✅ 必须 |
| **能被别人复现** | 其他人按照方案可以独立执行 | ✅ 必须 |
| **能形成可验收结果** | 有明确的"做成功了"的标准 | ✅ 必须 |
| **能沉淀为项目资产** | 执行后产生的文档、代码、流程可以被后续复用 | ⭐ 强烈建议 |

### 不适合进入方案库的想法

- 纯概念性想法，没有具体实施路径
- 依赖专有技术或内部资源，外部无法复现
- 需要超过 2 周才能验证的复杂方案（建议拆分为更小的 Blueprint）
- 没有明确验收标准的"探索性"想法
- 无法构建真实 Demo 的想法

## 质量标准检查清单

在提交方案前，检查以下项目：

- [ ] 原始想法被完整保留，没有被改写或删除
- [ ] 问题描述具体，包含场景和量化数据
- [ ] MVP 可以在 1-3 天内验证
- [ ] 所有步骤具体到可以执行（不需要读者"补充"）
- [ ] 提供了可直接复制粘贴的代码/Prompt/配置
- [ ] 技术架构图用文字描述清楚，没有模糊地带
- [ ] 验收标准可验证，不是"感觉好"
- [ ] 没有遗漏关键步骤（权限、配置、环境准备）
- [ ] 语言简洁，没有废话
- [ ] 适合技术背景和非技术背景的读者阅读
- [ ] **构建了真实 Demo**（可运行的原型，不是截图或描述）
- [ ] **Demo Pack 完整**（inputs/ prompts/ outputs/ app/ validation/）
- [ ] **复现步骤清晰**（别人可以照着复现）
- [ ] **验收记录完整**（Demo 构建后的验证结果）

## 常见错误

| 错误 | 表现 | 修正 |
|------|------|------|
| 缺少原始想法 | 方案看起来是"设计出来的"，不是"从真实需求出发的" | 补充原始想法段落 |
| 问题描述模糊 | "效率不高"、"体验不好" | 改为"每次手动操作需要 15 分钟，每天重复 5 次" |
| MVP 过大 | 需要 2 周以上才能验证 | 缩减到核心假设，1-3 天验证 |
| 步骤不具体 | "配置好环境" | 改为"运行 `pip install -r requirements.txt`" |
| 缺少 Prompt | 方案提到 AI 但没有给 Prompt 模板 | 补充可复制的 Prompt |
| 没有验收标准 | 不知道"做完了"是什么意思 | 定义明确的标准，如"输出包含 10 个字段" |
| 过度技术化 | 堆砌技术术语，没有解释 | 假设读者不懂，解释每个技术选择的原因 |
| 缺少人工确认点 | 方案没有标注需要人工决策的地方 | 添加"人工确认点"章节 |
| 缺少沉淀方式 | 方案没有说明执行后产生什么资产 | 添加"项目沉淀方式"章节 |
| **没有真实 Demo** | 只有方案描述，没有可运行原型 | 构建最小 Demo，保存到 `app/` |
| **Demo Pack 不完整** | 缺少 inputs/ prompts/ outputs/ validation/ | 补充完整目录结构 |
| **没有复现步骤** | 别人无法照着复现 | 编写清晰的复现说明 |

## 格式规范

### 文件命名

- Markdown 版：`docs/{slug}.md`
- HTML 版：`public/blueprints/{slug}.html` 和 `docs/blueprints/{slug}.html`
- Demo Pack：`demos/{slug}/`
- 数据条目：`data/blueprints.json`

### slug 规则

- 全部小写
- 单词用连字符分隔
- 不要包含数字序号（用 `customer-meeting-autonomous-build` 而不是 `002-customer-meeting`）
- 保持简洁，不超过 5 个单词

### 图片/图表

- 优先使用文字描述和 ASCII 图
- 如需要图片，放在 `public/assets/` 或 `demos/{slug}/assets/` 目录下
- 使用相对路径引用
