# 项目记忆型会议助手

> **英文名：** Project Memory Meeting Assistant
> **类型：** AI Workflow
> **难度：** 中等
> **Demo 周期：** 1-2 天
> **适合对象：** 项目负责人、研发团队、AI Agent 使用者
> **标签：** AI, 会议纪要, 知识库, 项目助理
> **状态：** 完成

---

## 一句话介绍

基于项目知识库，让 AI 不是简单记录"会上说了什么"，而是回答"这场会对项目意味着什么"，并输出可直接执行的会议纪要、决策记录和下一步行动。

---

## 原始想法

> 我把本地一个中型项目的知识库搭建好了。把 Hermes Agent 接入到知识库里边。今天上午在公司开了一上午的会，用飞书妙记。开完会以后，把飞书妙记转成文本，扔给 Hermes Agent。让它基于我的项目知识库生成会议纪要。出来的会议纪要把我震惊了，特别牛逼。简直就是项目里边的一个全能项目助手，它记的会议纪要比我们自己还懂自己。我坚信，这就是以后 AI workflow 的方式。我的一个目标就是把这套牛逼的方式给铺开到我们整体的团队乃至公司。狠狠的震惊一下他们。

## 背景与问题

### 当前场景

在项目管理中，会议是最常见的信息同步和决策场景。但传统的会议纪要存在以下问题：

- 记录的是"会上说了什么"，而不是"这对项目意味着什么"
- 缺少项目上下文，导致决策记录孤立
- 行动项与项目文档、代码状态脱节
- 重复开会，因为之前的决策没有沉淀到知识库

### 现有痛点

- **痛点 1：信息孤岛** — 会议记录、项目文档、代码仓库、Bug 追踪各自为政，没有人把它们连接起来
- **痛点 2：上下文丢失** — 新成员加入项目时，不知道之前的决策为什么是这样的
- **痛点 3：重复讨论** — 同样的议题反复讨论，因为之前的结论没有系统性地沉淀和关联
- **痛点 4：行动项失控** — 会议中的行动项散落在聊天记录、邮件、文档中，无法追踪完成情况

### 为什么需要解决

一个中型项目的知识库（文档、代码、历史记录）通常包含数百甚至数千条信息。人类不可能记住所有内容，但 AI 可以。如果 AI 能基于项目知识库理解会议内容，它可以：
- 自动关联相关文档和历史决策
- 指出与现有结论的冲突
- 建议遗漏的参与者或需要同步的模块
- 预测对项目时间表的影响

---

## 为什么这个想法重要

这个想法不是"做一个更好的会议纪要工具"，而是**重新定义会议在项目中的角色**。

传统会议：信息同步 → 人工记录 → 遗忘  
项目记忆型会议：信息同步 → AI 理解 → 自动关联知识库 → 生成决策记录 → 回写知识库 → 增强项目记忆

每一次会议都在增强项目的"记忆"，而不是增加信息噪音。

### 目标用户

| 用户角色 | 使用场景 | 价值 |
|---------|---------|------|
| 项目负责人 | 每周项目例会、里程碑评审 | 自动生成项目级洞察，减少手动整理时间 |
| 研发团队 Lead | 技术方案评审、代码评审会 | 自动关联技术文档和历史决策 |
| 产品经理 | 需求评审、用户反馈讨论 | 自动关联需求文档和用户反馈记录 |

### 使用场景

1. **场景 1：项目例会** — 每周团队同步，AI 基于项目知识库生成包含项目状态变化、风险预警的会议纪要
2. **场景 2：技术评审** — 评审新技术方案时，AI 自动关联现有架构文档和之前的技术决策
3. **场景 3：客户会议** — 与客户讨论需求变更时，AI 自动关联原始需求文档和已承诺的交付范围

## 方案概述

项目记忆型会议助手是一个 AI Workflow，连接三个核心组件：

1. **项目知识库** — 包含项目文档、代码、历史记录、决策日志
2. **会议转写** — 通过飞书妙记等工具将会议音频转为文本
3. **AI 项目助手** — 基于项目知识库理解会议内容，生成项目级会议纪要

核心差异：普通会议纪要回答"这场会说了什么"，项目记忆型会议助手回答"这场会对项目意味着什么"。

它不只是记录，而是**理解、关联、推理、建议**。

## Demo 最小可行版本（MVP）

### 目标

用 1-2 天验证核心假设：AI 基于项目知识库生成的会议纪要是否比人工整理的更有价值。

### 输入

1. **项目知识库** — 一个包含项目文档、代码 README、阶段报告的文件夹（或向量数据库）
2. **会议转写文本** — 飞书妙记导出的会议文本（或任何会议转写工具）
3. **固定 Prompt 模板** — 定义 AI 的理解框架和输出格式

### 输出

1. **项目级会议纪要** — 包含会议核心摘要、项目背景关联、决策记录
2. **决策记录** — 明确列出本次会议做出的决策
3. **行动项** — 具体的下一步行动，含负责人、截止时间、关联文档
4. **风险与阻塞** — 本次会议暴露的新风险或阻塞点
5. **对现有文档的影响** — 哪些现有文档需要更新或补充
6. **项目状态变化** — 本次会议对项目整体状态的影响
7. **下一步建议** — 给项目负责人的具体建议

### 验证标准

- [ ] AI 输出的会议纪要包含至少 3 个基于项目知识库的关联（如"这与文档 X 中的结论一致/冲突"）
- [ ] 行动项明确到可以执行（含"谁做什么"）
- [ ] 决策记录可以被后续会议引用
- [ ] 输出适合直接发给团队成员阅读，不需要二次整理

## 数据流 / 工作流

### 核心流程

```
项目文档/代码/阶段报告 → 项目知识库 → 飞书妙记会议转写 → Hermes Agent → 项目级会议纪要 → 人工确认 → 回写知识库 → 下一轮项目记忆增强
```

### 每个步骤说明

1. **项目知识库构建**
   - 收集项目文档、代码 README、阶段报告、历史决策
   - 将文本向量化，存储到向量数据库
   - 建立文档间的关联关系（可选）

2. **会议转写**
   - 使用飞书妙记录制会议（或导入录音）
   - 导出文本格式的会议转写
   - 清理格式，保留说话人标识（可选）

3. **AI 生成会议纪要**
   - 将会议转写文本输入 AI 项目助手
   - AI 从项目知识库检索相关文档（Top-K）
   - AI 基于会议内容 + 项目知识生成结构化输出

4. **人工确认**
   - 负责人审查 AI 生成的会议纪要
   - 修正事实错误
   - 确认行动项和负责人
   - 标记"待确认"项

5. **回写知识库**
   - 将确认的会议纪要存入知识库
   - 更新相关文档（如需要）
   - 将决策记录标记为"已确认"

6. **记忆增强**
   - 下一轮会议时，AI 可以引用之前的决策记录
   - 项目知识库持续增长，AI 理解越来越深

## 风险与限制

| 风险 | 影响 | 应对 |
|------|------|------|
| AI 幻觉 | 高 | 要求区分"会议内容"和"知识库推断"，不确定标记"待确认" |
| 知识库不完整 | 中 | 初期接受不完美，逐步补充文档 |
| 项目隐私泄露 | 高 | 使用本地 LLM 或确保 API 调用不泄露敏感信息 |
| 团队成员不接受 AI 输出 | 中 | 强调"辅助"而非"替代"，人工确认环节不可跳过 |
| 知识库维护成本 | 中 | 建立文档更新流程，确保知识库与项目同步 |

### 已知限制

- 当前方案依赖外部 AI API，可能产生费用
- 向量数据库的检索质量取决于文档质量和分块策略
- 需要人工确认环节，不能完全自动化
- 适合中型及以上项目，小项目可能知识库内容不足

## 后续扩展方向

1. **多模态输入** — 支持会议视频、PPT、白板内容的理解
2. **实时会议助手** — 会议进行中实时提供相关文档和决策提醒
3. **行动项追踪** — 自动追踪行动项完成情况，在下次会议前提醒
4. **跨项目关联** — 多个项目的知识库关联，识别跨项目依赖和影响
5. **智能会议摘要** — 根据参会人角色生成定制化摘要（技术视角、管理视角、客户视角）
6. **历史趋势分析** — 分析多次会议的趋势，预测项目风险
7. **集成项目管理工具** — 自动在 Jira/Linear/Notion 中创建任务

## 可直接复制给 Agent 的 Prompt 模板

```
你是当前项目的 AI 项目助手。

请基于以下会议转写文本和项目知识库，生成项目级会议纪要。

## 会议转写文本

[在此粘贴会议转写文本]

## 项目知识库（已检索相关文档）

[在此粘贴向量数据库检索到的 Top-K 相关文档]

## 请输出以下内容

1. **会议核心摘要** — 用 3-5 句话概括会议的核心议题和结论

2. **本次会议涉及的项目背景** — 基于项目知识库，列出与本次会议相关的项目背景信息（如相关模块、历史决策、当前状态）

3. **明确决策** — 列出本次会议做出的所有明确决策。格式：
   - 决策内容
   - 决策依据（引用知识库中的相关文档或会议中的讨论）
   - 影响范围

4. **待确认问题** — 列出会议中提出但尚未明确结论的问题。格式：
   - 问题描述
   - 需要谁确认
   - 建议的确认方式

5. **行动项** — 列出所有需要执行的行动项。格式：
   - 行动内容
   - 建议负责人（如果不确定，标记为"待确认"）
   - 建议截止时间（如果会议中提到）
   - 关联文档/代码/任务

6. **对现有项目文档的影响** — 基于知识库，分析本次会议是否会影响现有文档。格式：
   - 需要更新的文档
   - 需要新增的文档
   - 需要废弃或归档的文档

7. **对代码、产品、测试或交付的影响** — 分析本次会议对技术实现的影响。格式：
   - 代码层面：需要修改/新增的模块
   - 产品层面：需求变更、功能调整
   - 测试层面：测试范围变化
   - 交付层面：时间表、里程碑影响

8. **风险与阻塞** — 列出本次会议暴露的新风险或阻塞点。格式：
   - 风险/阻塞描述
   - 影响程度（高/中/低）
   - 建议应对措施

9. **项目状态变化** — 基于知识库，分析本次会议对项目整体状态的影响。格式：
   - 状态变化描述（如：从"按计划"变为"有风险"）
   - 变化原因
   - 需要关注的关键指标

10. **给项目负责人的下一步建议** — 基于以上分析，给出 3-5 条具体建议。格式：
    - 建议内容
    - 优先级（高/中/低）
    - 预期效果

## 要求

- **区分来源**：明确区分"会议明确提到的内容"和"基于知识库推断的内容"，后者用 [推断] 标记
- **不要编造**：不要编造负责人、时间、数据。如果不确定，标记为"待确认"
- **标记不确定**：所有不确定的信息用 [待确认] 标记
- **适合阅读**：输出适合直接发给团队成员阅读，语言简洁，结构清晰
- **引用知识库**：在适当位置引用知识库中的相关文档，增强可信度

## 输出格式

请用 Markdown 格式输出，包含标题、列表、表格等结构化元素。确保可以直接复制到飞书文档或 Notion 中使用。
```
