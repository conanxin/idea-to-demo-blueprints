客户会议触发的自主循环构建

AI Workflow / Agent Automation 难度：中高 Demo 周期：2-3 天 适合：产品经理 / 售前团队 / 创业团队

原始想法 / 灵感来源

客户在会议中描述需求，AI 在后台实时转录、理解、拆解，并触发 agent 构建可演示 Demo。到会议结束时，团队可以展示一个基于客户刚刚表达的工作流原型。

背景与问题

当前场景

客户会议是需求收集、方案沟通、产品验证的核心场景。但传统的工作流程存在严重的效率瓶颈：

  • 客户在会议中会产生大量需求、场景、抱怨、限制和隐含目标
  • 传统方式是会后人工整理，再排期，再做原型
  • 这个链条很慢，而且容易丢失客户刚刚表达的真实语境
  • 很多需求不是写在 PRD 里，而是藏在客户的描述、例子、犹豫和反复强调中

现有痛点

  • 痛点 1：需求丢失 — 会议中客户表达的 80% 细节在会后整理时丢失，特别是语气、强调和隐含限制
  • 痛点 2：整理周期长 — 从会议结束到可演示原型，通常需要 1-2 周，客户热情已经消退
  • 痛点 3：理解偏差 — 产品经理对客户需求的理解可能与实际意图存在偏差，但直到原型做出来才发现
  • 痛点 4：反馈循环慢 — 客户看到原型后提出修改，再次排期、再次开发，循环周期长
  • 痛点 5：售前竞争力 — 在竞争激烈的 ToB 市场，"会议结束就能演示原型"是极强的差异化优势

为什么需要解决

如果 AI 能在会议中实时理解需求，并在后台触发 Agent 构建原型，团队就可以极大缩短"客户表达 → Demo 验证"的周期。这不仅提升效率，更重要的是：

  • 在客户最兴奋的时候展示原型，最大化转化率
  • 让客户感到"被理解"，建立信任
  • 快速验证需求是否真实，避免投入大量资源后发现方向错误
  • 在竞争中建立"响应速度"的壁垒

为什么这个想法重要

普通客户会议产出的是会议纪要。

这个方案产出的是：

  • 结构化需求
  • 用户故事
  • 流程图
  • Demo 原型
  • 可运行页面
  • 下一轮客户验证材料

核心差异：

普通会议助手回答："客户说了什么？"

自主循环构建回答："客户刚才说的东西，能不能立刻变成一个可演示 Demo？"

这不是记录工具，而是实时构建引擎。它把客户的话语转化为可触摸的产品，把模糊的需求转化为清晰的原型。

使用场景

  1. 客户访谈 — 与潜在客户深入交流，AI 实时提取需求并构建原型，访谈结束时展示初步方案
  2. 售前方案沟通 — 销售团队与客户讨论需求，AI 在后台构建定制化 Demo，会议结束时展示
  3. 需求澄清会议 — 产品经理与业务部门对齐需求，AI 实时构建原型验证理解是否正确
  4. 内部产品评审 — 产品团队评审新功能，AI 快速构建交互原型供讨论
  5. MVP 探索 — 创业团队与早期用户交流，快速验证核心假设
  6. 黑客松式快速验证 — 研发团队在短时间内验证技术可行性

方案概述

核心流程：

客户会议 → 实时转录 → AI 提取需求和场景 → 生成结构化需求 brief → Agent 拆解任务 → Agent 构建 Demo → 会议结束前或会后短时间内展示原型 → 客户反馈 → 继续迭代

这个方案将客户会议从"信息收集环节"升级为"实时构建环节"。客户不是在"描述未来想要什么"，而是在"看着自己的需求变成现实"。

Demo 最小可行版本

目标

用 2-3 天验证核心假设：AI 能否从客户会议文本中提取需求并构建一个可演示原型。

第一版 Demo 策略

不需要一开始做复杂实时系统。第一版 Demo 可以用"准实时 / 会后快速构建"方式验证：

  • 会后 30 分钟内，将会议转写文本输入 AI
  • AI 生成结构化需求 brief 和 Agent 任务清单
  • Agent 在 1-2 小时内构建最小原型
  • 当天或次日向客户展示

输入

  1. 客户会议转写文本 — 飞书妙记、Zoom 等导出的会议文本
  2. 产品或项目背景文档 — 产品定位、技术栈、现有功能、限制条件
  3. Agent 构建 Prompt — 定义如何从需求 brief 生成可执行构建任务
  4. 简单前端模板或项目脚手架 — 用于快速构建原型（HTML/CSS/JS 或 Next.js/Vite）

输出

  1. 需求摘要 — 客户核心目标、痛点、约束的结构化总结
  2. 用户故事 — 基于客户描述生成的用户故事列表
  3. 工作流拆解 — 客户描述的流程转化为可执行步骤
  4. Demo 页面说明 — 原型页面结构、交互、数据流说明
  5. Agent 任务清单 — 可交给 Agent 执行的构建任务列表
  6. 可打开的 HTML 原型页面 — 一个可运行的 Demo 页面（静态或低代码）
  7. Demo 讲解稿 — 向客户展示原型时的讲解要点

验证标准

  • 能从会议文本中提取客户目标（不是提取关键词，而是理解意图）
  • 能识别痛点、约束和隐含需求
  • 能生成清晰用户故事
  • 能生成 Agent 可执行任务
  • 能构建一个可打开的 Demo 页面
  • Demo 能对应客户会议中的真实表达
  • 不确定内容会被标记为"待确认"
  • 能生成下一轮客户沟通材料
  • 页面能让别人照着复现这个 Demo 流程

数据流 / 工作流

核心流程

客户自然语言 → 转写文本 → 需求抽取 → 场景归纳 → 用户故事 → Demo Brief → Agent Task Plan → 原型构建 → 人工审查 → 客户演示 → 反馈回写

每个步骤说明

  1. 客户自然语言 — 客户在会议中描述需求、痛点、期望、约束。不是结构化输入，而是自然语言。
  2. 转写文本 — 使用飞书妙记、Zoom transcript、Whisper 等工具将会议音频转为文本。保留说话人标识和语气。
  3. 需求抽取 — AI 从转写文本中提取：客户目标、痛点、期望功能、约束条件、优先级暗示。区分"明确表达"和"隐含需求"。
  4. 场景归纳 — 将客户描述的具体场景归纳为可复用的用户场景。识别关键角色、关键流程、关键决策点。
  5. 用户故事 — 将场景转化为标准用户故事格式：作为 [角色]，我想要 [功能]，以便 [价值]。每个故事包含验收标准。
  6. Demo Brief — 生成结构化 Demo Brief，包含：页面结构、核心交互、数据模型、样式方向、非目标。
  7. Agent Task Plan — 将 Demo Brief 拆解为 Agent 可执行的任务清单：创建文件、编写组件、配置路由、添加样式。
  8. 原型构建 — Agent 执行构建任务，生成可运行页面。使用项目脚手架或模板加速。
  9. 人工审查 — 产品负责人检查原型：是否准确反映客户需求？是否有错误承诺？是否需要调整？
  10. 客户演示 — 向客户展示原型，讲解如何对应会议中的讨论。收集实时反馈。
  11. 反馈回写 — 将客户反馈、需求变更、优先级调整写回知识库，为下一轮构建做准备。

演示策略

  • 快速展示（5 分钟）：场景摘要 → 痛点 → 核心流程 → Demo 原型
  • 深度讨论（15 分钟）：用户故事 → 待确认问题 → 下一步建议 → 收集反馈
  • 技术说明（可选）：构建日志摘要 → 技术可行性 → 交付预估

实施步骤

步骤 1：准备客户会议样本文本（30 分钟）

  • 选择一次真实的客户会议录音（或模拟一段）
  • 使用飞书妙记等工具导出文本
  • 确保文本清晰，包含客户描述、问题、期望、约束

步骤 2：准备产品/项目背景文档（30 分钟）

  • 编写产品定位文档（1 页）：产品是什么、解决什么问题、目标用户
  • 列出技术栈和现有功能
  • 列出限制条件（不能做什么、不做什么）
  • 准备组件库或模板（如果有）

步骤 3：编写需求提取 Prompt（1 小时）

  • 参考下方 Prompt 模板，编写适合你的产品/场景的需求提取 Prompt
  • 测试 Prompt：输入会议文本，检查输出是否包含所有必需字段
  • 调整 Prompt，确保能识别隐含需求和约束

步骤 4：生成结构化 Demo Brief（30 分钟）

  • 将会议文本 + 产品背景 + Prompt 输入 AI
  • 检查生成的 Demo Brief：是否完整？是否准确？是否有"待确认"项？
  • 人工审查 Brief，标记需要调整的地方

步骤 5：让 Agent 基于 Demo Brief 生成任务计划（30 分钟）

  • 将 Demo Brief 输入 Agent（如 Claude Code、Hermes）
  • 让 Agent 生成任务清单：文件、组件、交互、数据
  • 检查任务清单：是否可执行？是否完整？

步骤 6：让 Agent 构建最小原型（1-2 小时）

  • 提供项目脚手架或模板
  • 让 Agent 执行任务清单
  • 监控构建过程，处理错误
  • 验证原型是否可打开、可交互

步骤 7：人工检查 Demo 是否符合客户表达（30 分钟）

  • 对照会议文本，检查原型是否准确反映客户需求
  • 标记"待确认"项
  • 检查是否有错误承诺（客户没说的功能）
  • 准备讲解要点

步骤 8：把会议、Brief、Demo 和反馈回写知识库（30 分钟）

  • 保存会议转写文本到知识库
  • 保存 Demo Brief（结构化需求）
  • 保存 Demo 版本（代码 + 截图）
  • 记录客户反馈（如果已展示）
  • 更新产品路线图或需求池

验收标准

功能验收

  • 能从会议文本中提取客户目标（不是提取关键词，而是理解意图）
  • 能识别痛点、约束和隐含需求
  • 能生成清晰用户故事
  • 能生成 Agent 可执行任务
  • 能构建一个可打开的 Demo 页面
  • Demo 能对应客户会议中的真实表达
  • 不确定内容会被标记为"待确认"
  • 能生成下一轮客户沟通材料
  • 页面能让别人照着复现这个 Demo 流程

质量验收

  • 区分"客户明确表达"和"AI 推断"，后者用 [推断] 标记
  • 不承诺正式交付，明确说明是 Demo 原型
  • 不编造客户没有说过的功能
  • 不确定内容标记为 [待确认]
  • 输出适合直接发给团队成员和 Agent 使用
  • 讲解稿适合向客户展示，不过度技术化

性能验收（可选）

  • 从会议结束到生成 Demo Brief，不超过 30 分钟
  • 从 Demo Brief 到构建原型，不超过 2 小时
  • 原型页面加载时间不超过 3 秒

风险与限制

风险影响应对
转录错误高保留原始音频，关键需求人工复核；使用高质量转录工具
客户表达模糊高标记为 [待确认]，准备澄清问题；不猜测
Agent 过度推断中严格区分"客户明确表达"和"AI 推断"；人工审查环节不可跳过
会议中不能展示未经确认的承诺高所有 Demo 明确标注"原型验证用，非正式交付承诺"
Demo 不是正式产品中客户沟通时明确区分 Demo 和正式产品；Demo 用于需求验证
需要区分"客户明确说过"和"AI 推断"高在 Brief 和 Demo 中标注来源；人工审查时重点检查
涉及客户数据时需要脱敏高处理客户数据前确认授权；敏感信息替换为占位符
实时构建对工程模板和 Agent 稳定性要求较高中第一版使用"会后快速构建"而非实时；准备多个模板方案
客户可能误将 Demo 当作正式承诺高在 Demo 页面显著位置标注"原型验证"；沟通时明确说明

已知限制

  • 当前方案依赖外部 AI API，可能产生费用
  • 原型质量取决于项目脚手架和组件库的成熟度
  • 复杂交互和动态数据需要更多人工调整
  • 不适合需要后端逻辑的原型（纯前端或静态页面）
  • 客户会议质量直接影响输出质量（客户表达越清晰，输出越好）
  • 涉及客户数据时需要隐私合规审查

后续扩展方向

  1. 实时会议监听 — 会议进行中实时转录、实时提取需求、实时构建原型
  2. 自动生成 PRD — 从会议文本和 Demo 自动生成产品需求文档
  3. 自动生成 Figma 原型 — 从需求 Brief 生成 Figma 设计稿

你是客户需求的 AI 分析助手，也是 Demo 构建的规划者。

请基于以下客户会议转写文本和产品背景，生成一个完整的 Demo Brief 和可执行构建任务。

## 客户会议转写文本

[在此粘贴客户会议转写文本]

## 产品/项目背景

[在此粘贴产品定位、技术栈、现有功能、限制条件]

## 请输出以下内容

### 1. 客户目标摘要

用 3-5 句话概括客户的核心目标。区分：
- 明确目标（客户直接说出的）
- 隐含目标（客户暗示或反复提及的）
- 约束目标（客户因为限制而不得不接受的）

### 2. 客户痛点

列出客户描述的所有痛点。格式：
- 痛点描述（用客户的语言）
- 痛点影响（对客户业务的影响）
- 当前 workaround（客户现在的临时解决方案）
- 期望解决方式（客户暗示的解决方向）

### 3. 用户故事

基于客户描述，生成用户故事。格式：
- 作为 [角色]
- 我想要 [功能/能力]
- 以便 [价值/结果]
- 验收标准：[具体可验证的标准]
- 优先级：[高/中/低]（基于客户语气和强调程度）

### 4. 关键工作流

描述客户描述的核心业务流程。格式：
- 步骤 1：[做什么] → [谁做] → [输出什么]
- 步骤 2：[做什么] → [谁做] → [输出什么]
- 决策点：[什么条件下走 A 分支，什么条件下走 B 分支]
- 异常流：[出错时怎么处理]

### 5. 待确认问题

列出会议中提出但尚未明确的问题。格式：
- 问题描述
- 为什么重要（如果不确认会有什么风险）
- 建议的确认方式（问客户什么、找谁确认）
- 优先级（高/中/低）

### 6. Demo 范围

明确这个 Demo 要展示什么、不展示什么。格式：
- 包含的功能（客户明确提到的）
- 不包含的功能（客户没提到或明确排除的）
- 模拟的数据（用占位符表示真实数据）
- 假定的权限（假设用户已登录/已授权）

### 7. 非目标

明确说明这个 Demo 不做什么，避免误解：
- 不是正式产品
- 不做真实数据集成
- 不做后端逻辑（如果是纯前端 Demo）
- 不承诺交付时间
- 其他非目标

### 8. 页面结构

设计 Demo 页面的结构。格式：
- 页面 1：[名称] → [功能] → [关键元素]
- 页面 2：[名称] → [功能] → [关键元素]
- 导航流：[从页面 A 到页面 B 的触发条件]
- 核心交互：[用户最重要的 3 个操作]

### 9. 数据模型

描述 Demo 中需要的数据结构。格式：
- 实体 1：[名称] → [字段] → [示例数据]
- 实体 2：[名称] → [字段] → [示例数据]
- 关系：[实体之间如何关联]
- 示例数据：提供 3-5 条真实感示例数据

### 10. Agent 构建任务

将以上分析转化为 Agent 可执行的任务清单。格式：
- 任务 1：[操作] → [文件/组件] → [预期输出]
- 任务 2：[操作] → [文件/组件] → [预期输出]
- 依赖关系：[任务 A 必须在任务 B 之前完成]
- 验收标准：[怎么判断这个任务做成功了]

### 11. 验收标准

怎么判断这个 Demo 构建成功了？
- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

### 12. 演示讲解稿

为产品经理/售前准备一份讲解稿，用于向客户展示 Demo。格式：
- 开场（30 秒）："根据刚才的讨论，我们快速构建了一个原型..."
- 场景对应（2 分钟）："您提到的 [痛点] 对应这个 [功能]..."
- 核心演示（3 分钟）："让我们看看这个流程..."
- 待确认（1 分钟）："这里有 [X] 个问题需要确认..."
- 下一步（30 秒）："基于您的反馈，我们下一步可以..."

### 13. 下一轮客户沟通问题

基于 Demo 和当前分析，生成下一轮客户沟通的问题清单。格式：
- 确认类问题：[需要客户明确 Yes/No 的问题]
- 优先级问题：[需要客户排序的问题]
- 技术问题：[需要客户确认技术约束的问题]
- 业务问题：[需要客户补充业务背景的问题]

## 要求

- **区分来源**：明确区分"客户明确表达的内容"和"基于产品背景推断的内容"，后者用 [推断] 标记
- **不要承诺**：不要承诺正式交付时间、功能完整性、数据准确性。明确说明这是 Demo 原型
- **不要编造**：不要编造客户没有说过的功能、数据、用户角色。如果不确定，标记为 [待确认]
- **标记不确定**：所有不确定的信息用 [待确认] 标记
- **Demo 定位**：输出明确说明这是"需求验证原型"，不是"正式产品承诺"
- **适合 Agent**：输出要适合直接交给 Agent（如 Claude Code、Hermes）继续构建
- **适合客户**：讲解稿要适合向客户展示，不过度技术化
