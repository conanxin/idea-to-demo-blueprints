设计理由

为什么这个方案是必要的？

"手势实时视觉"是 2026 年社交媒体上的爆款技术叙事。一双手在摄像头前快速变换动作，屏幕上就跟着出现夸张的 3D 故障视觉——画面挤压、扭曲、闪烁、色彩切换，看起来像把 Figma 和 Photoshop 打垮了一样。

这个叙事很有传播力，但它有一个致命的副作用：观众看到的是结果，但不知道底层链路。真实链路是：

摄像头输入 → MediaPipe Hand Landmarker 检测 → 双手 21 个关键点 → 手势语义计算（中点 / 距离 / 角度 / 捏合 / 速度 / 握拳） → TouchDesigner 参数映射 → 实时 3D 故障视觉输出

本 Blueprint 的设计理由是：把这条链路结构化、可记录、可复现、可教学。视频带来灵感，但 Blueprint 带来技能。

为什么采用"Web 模拟器 + TouchDesigner 真机模板"双轨设计？

  • Web 模拟器 — 让没有 TouchDesigner 的人也能拖动手点、看视觉变化、读懂映射规则。GitHub Pages 静态部署即可访问。
  • TouchDesigner 真机模板 — 提供 Python DAT Prompt 模板和节点图规划，让本地有 TouchDesigner 的用户能复现真实效果。
  • 不提交 .toe 二进制 — TouchDesigner 工程文件是二进制且体积大，不适合版本控制。用户按 references 复制节点结构即可。

为什么不用真实摄像头做 Web Demo？

  • 隐私考虑 — GitHub Pages 静态部署，请求摄像头权限会引发用户警觉，且 MediaPipe WASM 在浏览器内运行对性能有要求
  • 演示可控性 — 拖动手点 + slider 可以精确演示每项映射，让讲解更可控
  • 移动端兼容 — 真实摄像头方案在移动端不可靠，拖动手点方案移动端可用
  • 真机方案已存在 — 真实摄像头方案在 TouchDesigner 端由 MediaPipe 插件完成，Web 端定位是"解释映射逻辑"

背景与问题

当前场景

创意技术、互动装置、舞台视觉、VJ 表演领域一直存在一个共同痛点：实时视觉需要"复杂工具 + 复杂输入"。常见工作流：

  • 用 TouchDesigner / Notch / Resolume 搭建视觉管线
  • 用 MIDI 控制器、Kinect、Leap Motion、自定义硬件做输入
  • 把两者联调需要懂节点图、信号流、Python / GLSL

这导致一个尴尬的局面：真正"懂视觉"的设计师不一定会写节点图，会写节点图的技术人不一定有表演感的肢体表现力。

现有痛点

  • 痛点 1：工具门槛高 — TouchDesigner 非商用版虽然免费，但节点图 + Python DAT + CHOP 信号流的组合学习曲线陡峭
  • 痛点 2：手部追踪方案分散 — MediaPipe Hands、Hand Landmarker、Leap Motion、Kinect 各有优缺点，选型没有标准答案
  • 痛点 3：手势到视觉的映射关系藏在脑子里 — 设计师凭直觉做映射，但没有结构化记录
  • 痛点 4：AI 辅助脚本缺位 — TouchDesigner Python DAT 可以写，但 prompt 和上下文都不容易凑齐
  • 痛点 5：缺少可解释的 Demo — 观众看到的是"哇好酷"，但创作者需要向同行、老师、客户解释每一帧视觉变化对应哪个手势

方案概述

这个方案的核心是把"摄像头 → MediaPipe → 手势语义 → TouchDesigner → 实时视觉"这条链路结构化、可记录、可复现。它由三层组成：

  1. 真实运行层（TouchDesigner 端） — 在本地 TouchDesigner + MediaPipe 插件环境下运行
  2. Web 交互模拟器（GitHub Pages 端） — 用 HTML/CSS/JS 复现手势映射逻辑，让没有 TouchDesigner 的人也能理解
  3. AI 辅助脚本层（Claude / Codex / Agent 端） — 用 prompt 模板让 AI 生成 TouchDesigner Python DAT、MediaPipe 集成代码、Web Demo 代码

MVP（Gesture Cube Lite）核心手势映射：

双手中点 → cube 位置 双手距离 → cube 缩放 双手连线角度 → cube 旋转 左手捏合 → 噪点 / distortion 强度 右手捏合 → 切换颜色主题 快速移动 → 触发 glitch 双手握拳 → freeze frame

Web Demo 定位明确：这是静态 Web 模拟器，验证手势映射逻辑和 Demo 可解释性。真实 TouchDesigner 版本需要在本地安装 TouchDesigner + MediaPipe 插件。

Demo MVP

目标

用 2-3 天验证："手势 → 视觉"映射规则能不能被结构化记录、复现、讲解？不要求做到完整的 TouchDesigner 真机工程，但要求：

  • Web 交互模拟器可拖动、可观察视觉变化
  • 映射规则和参数面板完整可见
  • TouchDesigner Python DAT 脚本有完整模板，可在本地真实环境复现

  • Agent 输入 — 固定的 Prompt 模板 + 当前手势映射规则文档 + 项目脚手架
  • 输出 — Web 端 cube 实时视觉 + TouchDesigner 真机视觉 + 沉淀产物（映射规则文档、Python DAT 脚本、Web Demo 源码）

  • 双手角度变化 → cube 旋转

  • 参数面板实时显示 9 项核心参数

数据流 / 工作流

核心流程

手势输入（拖动 or 摄像头） → 手部关键点检测（MediaPipe） → 手势语义计算（中点 / 距离 / 角度 / 捏合 / 速度 / 握拳） → 视觉参数映射（cube 位置 / 缩放 / 旋转 / 噪点 / 主题 / glitch / freeze） → 视觉输出更新（CSS 3D 或 TouchDesigner 节点图） → 参数面板实时显示 → （可选）AI Agent 辅助生成 / 修改映射规则

每个步骤说明

  1. 手势输入 — Web 端用鼠标拖动两个手点（模拟双手位置）；TouchDesigner 端用摄像头 + MediaPipe 检测真实手部
  2. 手部关键点检测 — MediaPipe Hand Landmarker 输出每只手的 21 个关键点坐标 + 置信度
  3. 手势语义计算 — 从 21 个关键点计算：双手中点（取 wrist 平均）、双手距离、双手连线角度、捏合度（拇指-食指距离归一化）、移动速度（连续帧差）、握拳状态
  4. 视觉参数映射 — 每个手势语义量映射到具体的视觉参数。映射规则用纯函数实现
  5. 视觉输出更新 — Web 端用 CSS transform / filter / class 切换；TouchDesigner 端用 CHOP 信号传到节点图参数
  6. 参数面板实时显示 — Web 端有 9 项参数面板
  7. AI 辅助 — 用 prompt 模板让 Claude / Codex 生成 / 调整 TouchDesigner Python DAT 脚本或 Web Demo 代码

关键 Prompt

Prompt A：生成 TouchDesigner Python DAT 脚本

你是 TouchDesigner + MediaPipe 集成专家。请基于以下手势映射规则，生成一个
完整的 TouchDesigner Python DAT 脚本（mediapipe_hand_callbacks.py），用于
GestureVisual Lab Blueprint。

## 手势映射规则（必须严格按此映射）

1. 双手中点 (center_x, center_y) → cube 位置 (translate_x, translate_y)
2. 双手距离 (distance) → cube 缩放 (scale)
3. 双手连线角度 (angle) → cube 旋转 (rotation_z)
4. 左手捏合 (left_pinch) → 噪点强度 (noise_amount)
5. 右手捏合 (right_pinch) → 主题切换（neon / mono / cyber）
6. 移动速度 (speed) → glitch 闪烁
7. 双手握拳 (fist_left && fist_right) → freeze

## 要求

1. 先输出脚本结构说明，再写代码
2. 代码用函数化结构（getCenter, getDistance, getAngle, getPinch, getSpeed, getFist, mapRange）
3. 每个函数加简短注释
4. 在脚本开头说明哪些行需要用户替换（用户本地 .toe 节点名）
5. 不要使用任何真实摄像头设备，只用 mock 数据测试
6. 不要硬编码任何 API key / secret

风险影响应对

  • AI Agent 辅助脚本可能需要多轮迭代 — TouchDesigner Python DAT 与 MediaPipe 集成的细节多
