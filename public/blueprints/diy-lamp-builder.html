设计理由

为什么这个方案是必要的？

独立开发者和小品牌文创团队想做实体产品时，最常见的死法：每个 SKU 从零设计、外包打样一次性消耗、LED 灯带贴塑料外壳散热失败、没有任何"可定制"的中间形态。设计-制造-销售链路里，没有任何一个"想法 → 可演示原型"的快速通道。

本 Blueprint 的设计理由是：用受控定制（controlled customization）模式，把"不变的内核"做成强 IP，把"变的部分"做成可定制项。参考 Gantri / Framework Laptop / Local Motors 的思路，但聚焦在更小、更轻的"阅读灯"赛道。

为什么采用"固定内核 + 可变外壳"双层架构？

  • ReadingCore-01 永远是同一个内核 —— 灯芯模块、光电参数、装配接口都不变。所有 SKU 共享工程。
  • Shell 永远可变 —— 外壳只承担造型与文化表达。文创团队可以快速出多款风格 / 颜色 / 刻字。
  • 结构隔离 —— LED 灯带不直接贴塑料外壳；先贴铝型材散热件，再加乳白扩散罩，最后装进可定制外壳。从根本上避免灯带直贴塑料的发黄陷阱。
  • Demo 可在网页里完整表达 —— 即使没有真实 3D 打印，单页 HTML 已经能讲清楚"内核是什么、外壳怎么变、装配顺序是什么"。

原始想法

这一段是用户在聊天里说出来的原始想法，未经润色：

我想要做一个"北京风格"的阅读台灯：
  • 适合放在书桌旁边，晚上看书用
  • 外壳可以是可定制的，比如不同的风格、颜色、纹样、刻字
  • 但灯芯我想做成一个固定的、稳定的模块：24V 线性 LED，足够亮但不能刺眼
  • 我希望这个灯不要像网红灯具那样把灯带直接贴在塑料外壳上 —— 那样散热很差，时间长了塑料会发黄
  • 我想要的真正结构是：LED 灯带先贴到铝型材/铝槽上，前面再加一片乳白扩散罩，再整体装进那个可定制外壳
  • 外壳只承担造型，不参与散热，也不接触电气
  • 灯芯模块我叫它 ReadingCore-01，所有产品都用同一个内核，只是换外壳
参考一下 Gantri 那种"受控定制"的思路，但比它更小、更轻、更像阅读灯而不是氛围灯。

背景与问题

实体产品 Demo 缺失的关键中间环节：

  • 每个 SKU 都从零设计 —— 成本爆炸、迭代极慢
  • 外包打样一次性消耗 —— 几百块一个样品，错了就报废
  • 结构错了重来 —— LED 灯带贴塑料外壳、电源藏不进底座、散热不够外壳发黄
  • 没有任何"可定制"的中间形态 —— 要么全定制，要么完全标准款
  • 现有"配置器"只做配色 —— Nike By You、自行车配置器都没有架构推理

现有受控定制参考

Gantri — 3D 打印灯厂，核心工程一次做好，只换表面 Framework Laptop — 主板固定，端口/键盘/外壳可换 Bourbon Barrel — E26 灯座 + 通用 LED + 回收木桶外壳 Local Motors — 动力总成固定 + 车身可换

共同点：把"不变的部分"做成强 IP，把"变的部分"做成可定制项。

方案概述

DIY Lamp Builder 是一个纯静态单页 Demo，由 4 个区组成：

  • A. Idea Input —— 一句话产品想法输入
  • B. AI Analyze —— 自动产品架构（ReadingCore-01 固定 / Shell 可变）
  • C. Configurator —— 风格 / 颜色 / 刻字 + SVG 实时预览
  • D. Manufacturing Plan —— 实时 JSON 输出 + 装配步骤

整个 Demo 不依赖任何外部资源（无 CDN、无 build step、无 Three.js），用 HTML/CSS/JS 直接表达"实体产品想法 → 受控定制 Demo"的完整链路。

Demo MVP

目标：用 1-2 天时间，做出能让"产品小白" 5 分钟看懂"ReadingCore-01 + 可定制外壳"架构的可交互 Demo。

第一版策略

  • 不做真实 LLM 调用 —— AI Analyze 区使用轻量规则解析器，把关键词映射到产品决策；真实场景可复用 Prompt 模板接入 LLM。
  • 不做真实 CAD —— SVG mock 预览足够
  • 不做真实采购/支付 —— BOM 估算给范围即可
  • 不做真实光路计算 —— 500-800 lm 是设计目标，不是实测

关键 Prompt

完整 Prompt 见 demos/diy-lamp-builder/prompts/build-prompt.md，下面是精简核心：

你是「DIY 可定制阅读台灯构建器」的方案 Agent。 # 用户原始想法（粘到这里） {{USER_IDEA}} # 你的任务 请基于用户的想法，按以下四步生成产物： 1. Product Architecture（产品架构推理） 识别输入想法，把它拆成两层结构： - 固定灯芯模块：默认命名 ReadingCore-01 - 可定制外壳（Shell）：包含 Style / Color / Engraving 三类自由度 2. Brief（方案 Brief） 300-500 字，包含定位、人群、价值主张、关键风险。 3. Task Plan（任务计划） 按 [Idea Input → AI Analyze → Configurator → Manufacturing Plan] 四步列出。 4. Manufacturing Plan JSON 按 README 中规定的字段输出。 # 约束 - 灯芯模块不可编辑 - 外壳只承担造型，不承担散热和电气 - 输出必须是结构化 JSON - 不要给真实采购链接，不要给真实 CAD/STL

风险与限制

风险影响应对
AI Analyze 区是 mock 不是真 LLM中在 generated-demo-notes.md 中明确标注；真实场景复用 Prompt 模板
SVG 预览精度有限中仅作示意；如需高精度可后续接 Three.js 或真实 CAD
BOM 估算成本不准确中输出动态 BOM range，例如 $58-126 prototype；具体范围随 shell style、颜色涂装和 mock base/arm 变化，明确免责声明。
散热设计未在 Demo 中验证低结构图中明确"铝槽是散热通道"，建议实际打样做热测试
风格 / 颜色只有 4 款低Demo 演示架构正确性，款式扩展留给后续版本

已知限制

  • 不支持自定义 Style（仅 4 款预定义）
  • 不支持上传图片 / 文件
  • 不支持保存 / 分享配置（仅 Copy JSON）
  • 装配步骤是英文（不针对中文用户改写）
  • BOM 价格是估算范围，不根据地区浮动

  1. 更多 Style —— 把 4 款外壳扩展到 8 / 12 款，加入材质选项（PETG / PLA+ / 树脂 / 木材）

  3. 真实元器件 sourcing —— 给出可采购的 SKU 和替代方案
  4. 真实亮度/照度测试 —— 用 lux meter 验证 35-45 cm 处的 desk lux
  5. 更多 Color —— 加入 Pantone 色卡导入、自定义 hex
  6. 装配动画 —— SVG 中做分步装配动画，每步高亮一个部件
  7. 配置保存 / 分享 —— URL hash 编码当前配置，方便分享链接
  8. 多 ReadingCore 变体 —— ReadingCore-02（高显色 95+）、ReadingCore-03（带 USB-C 充电口）
  9. 真实 LLM 接入 —— 把 Idea Input 接到 Claude / GPT，按 build-prompt.md 输出架构和 JSON
  10. 与本 IDB 库联动 —— 用「项目记忆型会议助手」管理 ReadingCore-01 迭代历史；用「多 Agent 项目管理面板」管理多个 SKU 状态
